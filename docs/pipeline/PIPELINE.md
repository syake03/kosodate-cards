# PIPELINE.md — カード自動生成パイプライン（2役分担 / reviewまで自動・publishは人間）

このドキュメントは、カードを「品質を保ったまま半自動で量産する」ための運用手順書。
CONTENT_SPEC.md v1.2（作り方の詳細ルール）を前提に、**それを誰が・どの順で・どう検証しながら回すか**を定める。

---

## 基本思想：作る人とチェックする人を分ける

人間の編集部が「記者」と「校閲」を分けるのと同じ。1体のAIに作らせて信じるのではなく、
**生成役（Writer）と検証役（Reviewer）を別セッション（できれば別モデル）に分離**する。
CONTENT_SPEC.md は「調査ログを見ればカードの良し悪しが判定できる」設計なので、この分離と相性が良い。

品質は3つの層で守る：
1. **機械チェック層（自動・無判断）** … スキーマ違反・出典本数・禁止ドメインを機械的に弾く（§3）
2. **生成役 Writer（自動）** … CONTENT_SPEC通りに調査ログ＋カードを作る（§4）
3. **検証役 Reviewer（自動）** … Writerとは別セッションが第三者として採点・出典抜き打ち確認（§5）

この3層を通ったものだけが `status: review` になる。**`published` への昇格だけは人間が行う（§6）。**

---

## 1. 状態の定義（statusフィールド）

| status | 意味 | 誰が設定 |
|---|---|---|
| `draft` | Writerが作成中、または機械チェック・Reviewerで不合格 | Writer / 自動 |
| `review` | 3層すべてを通過。人間の最終確認待ち | Reviewer（合格時のみ） |
| `published` | 人間が確認し公開OKとしたもの。サイトに表示 | **人間のみ** |

**重要：サイトのトップ・一覧に表示するのは `published` のみ**にする（SPEC.md の表示条件を review→published に変更）。
これにより「自動生成されただけの未確認カードが公開される」事故を構造的に防ぐ。

---

## 2. 1バッチの流れ（1枚ずつ、この順で回す）

```
[人間] 作るカードをキューから指定（例：次の3枚）
   ↓
[機械] 既存カードのidと重複しないか確認
   ↓
[Writer] 調査ログ→カード生成（CONTENT_SPEC §1〜§8）→ status: draft で保存
   ↓
[機械チェック] スキーマ・出典本数・禁止ドメイン検査（§3）
   ├─ 不合格 → draftのままTODO記載、次へ or 差し戻し
   └─ 合格 ↓
[Reviewer] 別セッションが採点＋出典抜き打ち（§5）
   ├─ 不合格 → status: draft に戻し、指摘をカード末尾に記録
   └─ 合格 → status: review に昇格
   ↓
[人間] reviewキューを見て publish 判断（§6）
```

---

## 3. 機械チェック層（Claude Codeにスクリプト化を依頼）

`scripts/validate-cards.mjs` を作り、ビルド前とバッチ後に走らせる。**判断を要さない違反だけ**を検出する：

- フロントマター必須項目の欠落（id/title/statement/domain/ages/evidence/impact/tags/updated/status）
- `domain` が10種のenum外／`ages` が8種のenum外／`evidence` が A|B|C|observational 外／`impact` が critical|serious|qol 外
- `id` がファイル名と不一致、または既存idと重複
- 本文に `## 出典` セクションがない、または出典リンクが2本未満
- `impact: critical` なのに出典リンクが2本未満（criticalはTier1×2本以上が原則のため、最低ラインとして総数2本を機械チェック）
- 出典URLに**禁止ドメインパターン**が含まれる（ブログ/まとめ系の簡易ブラックリスト：`ameblo.jp` `note.com` `hatenablog` `fc2` `livedoor` 個人商用クリニック等。完全ではないが明白なものを弾く）
- `confidence_note` が200字超

出力は「カードごとに合格/不合格＋理由」の一覧。不合格は自動で `status: draft` に落とす。

> 機械チェックはあくまで足切り。「出典の中身が本当に主張を支持しているか」は判断できないので、そこはReviewerの仕事。

---

## 4. 生成役 Writer への指示（WRITER_PROMPT.md を使用）

別ファイル `WRITER_PROMPT.md` の内容をそのまま Writer セッションに渡す。要点：
- CONTENT_SPEC.md §1〜§8 に厳密に従う
- 調査ログを先に、カードを後に
- 出典は実際にfetchして確認したものだけ
- 判断に迷ったら低い方・書かない方に倒す
- 必ず `status: draft` で出す（自分でreviewにしない）

---

## 5. 検証役 Reviewer への指示（REVIEWER_PROMPT.md を使用）

別ファイル `REVIEWER_PROMPT.md` の内容を、**Writerとは別のセッション（別モデル推奨）**に渡す。要点：
- Writerが作った調査ログとカードを受け取り、**第三者として** CONTENT_SPEC §10 の12項目を1つずつ採点
- **出典URLを2本、実際に開いて**、カード本文の主張がその出典に本当に存在するか確認（抜き打ち）
- 数値を1つ選び、出典内に存在するか確認
- エビデンス・影響度の決定木のたどりが妥当か検証（特にcriticalの死亡裏付け、Aの根拠）
- 1つでも問題があれば `status: draft` に戻し、**具体的な指摘**をカード末尾にコメントで残す
- すべて合格した場合のみ `status: review` に上げる
- **Reviewerは「甘くしない」ことが仕事。** 迷ったら不合格にして人間に回す。

---

## 6. 人間の publish 判断（最後の砦・ここだけ手動）

Reviewerが `review` に上げたカードについて、人間は以下だけ確認すればよい（全文精読は不要）：

1. **タイトルとstatementを読む** … 推奨内容に違和感がないか
2. **confidence_noteを読む** … 留保が正直に書かれているか
3. **出典欄を見る** … 一次資料（go.jp / who.int / aap.org / cdc.gov 等）が並んでいるか
4. 気になったカードだけ、Reviewerの採点コメントと調査ログを確認

問題なければ `status: published` に変更（＋ `updated` を確認）。これでサイトに表示される。

> この6ステップのうち、人間が関与するのは「①最初にキューを指定」と「⑥publish判断」の2点だけ。
> 生成・機械チェック・検証は自動で回る。**公開の最終責任だけ人間が持つ。**

---

## 7. どのツールで各役をやるか（推奨）

| 役 | 推奨ツール | 理由 |
|---|---|---|
| 機械チェック | Claude Code（スクリプト実行） | ビルド環境と同じ場所で回せる |
| Writer | Cowork または Claude Code のセッション | Web検索・fetchができる |
| Reviewer | **Writerとは別のセッション**（このチャット、別のCoworkセッション等） | 独立性が命。同じセッションで自己採点させない |
| publish | 人間（GitHubでstatus書き換え→push） | 公開責任 |

**最小構成でよければ：** Writer=Cowork、Reviewer=このチャット（私）、publish=あなた、でも回る。
慣れてきたら Reviewer も自動セッション化する。

---

## 8. バッチサイズと停止ルール

- 1バッチ **3〜5枚**を上限にする（10枚一気は検証が甘くなる。実証済みの教訓）。
- Writerは1枚ごとに調査ログを完成させてから次へ。
- Reviewerが2枚連続で「同じ種類の不合格」を出したら停止し、人間に相談（Writer側かSPEC側に問題がある兆候）。
- キューを飛び越えない。派生トピックはTODOにメモのみ。

## 8.5 Claude Code でのサブエージェント実装（Reviewerも自動化する本命構成）

Claude Code の**サブエージェント**を使うと、Writer と Reviewer を「それぞれ独立した文脈ウィンドウ」で動かせる。これが独立性を構造的に保証する（同じ会話の中で自己採点させない、という最重要条件を機械的に満たす）。メイン session が全体のオーケストレーター（司令塔）になり、1枚ずつループを回す。

### 独立性がなぜ保証されるか
Claude Code のサブエージェントは、それぞれ独自の context window・独自のシステムプロンプト・独自のツール権限を持ち、**親には最終結果だけを返す**（作業途中の思考や検索結果は親やほかのサブエージェントに漏れない）。だから Writer が書いた思考プロセスは Reviewer の文脈に入らず、Reviewer は「まっさらな目」でカードと調査ログだけを読む。人間の校閲が原稿だけ見て著者の意図を知らないのと同じ状態を作れる。

### 3つのサブエージェントを定義する
プロジェクト直下に `.claude/agents/` を作り、以下3つを Markdown で定義する（frontmatter に name・description・tools・model、本文にシステムプロンプト）。

1. **`card-writer`**（生成役）
   - tools: 検索・fetch・ファイル読み書き（Read, Write, WebSearch, WebFetch 等）
   - 本文システムプロンプト = `WRITER_PROMPT.md` の中身を貼る
   - 必ず `status: draft` で出す

2. **`card-reviewer`**（検証役）
   - **tools: 読み取り＋検索・fetchのみ（Write/Edit を与えない）** ← これが肝。レビュー中にカードを勝手に書き換えられない。指摘はメインに返し、statusの変更はメインが行う。
   - 本文システムプロンプト = `REVIEWER_PROMPT.md` の中身を貼る
   - 出典を実際に開いて抜き打ち確認し、合否と指摘を**テキストで親に返す**

3. **`card-validator`**（機械チェック役・任意）
   - tools: Bash のみ（`scripts/validate-cards.mjs` を実行するだけ）
   - §3 の機械チェックを走らせ、結果を返す

> reviewer に書き込み権限を与えないのは、公式にも推奨される「reviewer は read-only、fixer が別」というパターン。校閲役が親切心で勝手に直してしまうと独立性が崩れるため、Claude Code のツール権限機能でそれを構造的に禁じる。

### メイン session（オーケストレーター）への依頼文
Claude Code の通常セッションで、以下を伝える：

```
このプロジェクトで PIPELINE.md の仕組みを実装したい。
.claude/agents/ に card-writer / card-reviewer / card-validator の3つのサブエージェントを作って。
- card-writer は WRITER_PROMPT.md をシステムプロンプトに、検索・fetch・書き込みを許可。
- card-reviewer は REVIEWER_PROMPT.md をシステムプロンプトに、【読み取りと検索・fetchのみ・書き込み禁止】。
- card-validator は scripts/validate-cards.mjs を実行するだけ。

作ったら、次のループを1枚ずつ回して：
1. 私が指定したトピックを card-writer に渡してカードと調査ログを draft で生成
2. card-validator で機械チェック。不合格なら draft のまま指摘を記録して次へ
3. 合格したら card-reviewer に渡して検証（出典の抜き打ち確認を含む）
4. reviewer が合格を返したらメイン（あなた）が status を review に変更
   reviewer が不合格を返したら draft のままカード末尾に指摘を記録
5. 1バッチ3〜5枚で止めて、review になったカードの一覧を私に見せて

重要：card-writer と card-reviewer は必ず別サブエージェント（別文脈）で動かし、
文脈を共有させないこと。publish への昇格は絶対に自動化せず、私（人間）がやる。
```

### この構成の効果
- Writer と Reviewer が別文脈なので、**自己採点にならない**（独立性が機械的に保証される）。
- Reviewer は read-only なので、**検証中にカードを書き換えられない**（判定と修正の混同を防ぐ）。
- メイン session が status 変更を握るので、**昇格の制御が1か所に集約**される。
- publish は依頼文で明示的に「人間がやる」と禁止しているので、自動で公開されない。

### 現実的な注意
- サブエージェントを多用すると**トークン消費が増える**（各自が独自文脈を持つため、単一スレッドの数倍になりうる）。1バッチ3〜5枚の上限はコスト面でも妥当。
- 自動 Reviewer は「出典を開いて記載を確認する」までは確実にできるが、「その分野で本当にこの機関が最も権威あるソースか」といった領域の土地勘は人間ほど鋭くない。だから **publish 前に人間がサマリー確認する最後の一手（§6）は残す**。ここは自動化しない。

---

## 9. まとめ：この仕組みの効果

- **Writerが手を抜いても、別のReviewerが出典を実際に開いて確認するので、質の落ちたカードはreviewに上がれない。**
- 機械チェックが明白な違反を先に弾くので、Reviewerは「中身の判断」に集中できる。
- 人間はキュー指定とpublish判断だけ。手間は最小、でも公開責任は手放さない。
- 「reviewまで自動・publishは人間」は、健康・安全情報を扱うこのプロジェクトの倫理に沿った線引き。
